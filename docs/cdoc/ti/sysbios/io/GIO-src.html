    48    /*!
    49     *  ======== GIO ========
    50     *  General Purpose IO Manager.
    51     *
    52     *  The GIO Manager offers both the issue/reclaim model and the read/write
    53     *  model to send and receive data from drivers that implement the IOM
    54     *  interface as provided in ti/sysbios/io/iom.h. 
    55     *
    56     *  In the issue/reclaim model, the client calls {@link #issue} when he has
    57     *  a buffer of data.  issue() is non-blocking and returns -- it simply
    58     *  submits the buffer to the driver for I/O.  The client calls
    59     *  {@link #reclaim} to get the buffer back.  A call to reclaim() may block.
    60     *  Upon return from reclaim(), the client can re-use the buffer. 
    61     *
    62     *  The client can issue many buffers before reclaiming them.
    63     *  Buffers are always reclaimed in the order that they were issued.
    64     *  The client can optionally pass a user argument to issue().  This argument
    65     *  can be retrieved with reclaim().
    66     *
    67     *  In the read/write model, clients will call {@link #read} or {@link #write}
    68     *  to send/receive data. Here the client may block until buffer is ready or 
    69     *  a timeout occurs.
    70     *
    71     *  The GIO module also provides {@link #control} to send down driver specific
    72     *  control  commands.  The {@link #abort} function can be used to abort any
    73     *  pending I/O.
    74     *
    75     *  The {@link ti.sysbios.io.DEV} module maintains a name table of IOM drivers.
    76     *  This table is used by GIO to create an IO stack. The name passed to
    77     *  {@link #create} is usually of the form "/uart". This name may
    78     *  correspond to the following IO stack.
    79     *
    80     *         GIO Instance
    81     *
    82     *              |
    83     *              v
    84     *
    85     *         IOM Instance (/uart)
    86     *
    87     *  The GIO module uses the {@link xdc.runtime.knl.Sync} module for
    88     *  synchronization.  GIO will call {@link xdc.runtime.knl.Sync#signal} when 
    89     *  I/O completes and {@link xdc.runtime.knl.Sync#wait} to wait for I/O.
    90     *
    91     *  By default the I/O manager will create a semaphore for synchronization
    92     *  if no sync handle is passed to {@link #create}. 
    93     */

   227        /*!
   228         *  ======== abort ========
   229         *  Abort all pending IO.
   230         *
   231         *  An application calls GIO_abort to abort all input and output from
   232         *  the device. When this call is made, all pending calls are completed
   233         *  with a status of GIO_ABORTED. An application uses this call to
   234         *  return the device to its initial state. Usually this is done in
   235         *  response to an unrecoverable error at the device level.
   236         *
   237         *  GIO_abort returns IOM_COMPLETED upon successfully aborting all
   238         *  input and output requests. If an error occurs, the device returns a
   239         *  negative value.  The list of error values are defined in the 
   240         *  ti/sysbios/io/iom.h file.
   241         *
   242         *  A call to GIO_abort results in a call to the mdSubmit function of the
   243         *  associated mini-driver. The IOM_ABORT command is passed to the
   244         *  mdSubmit function. The mdSubmit call is typically a blocking call, so
   245         *  calling GIO_abort can result in the thread blocking.
   246         *
   247         *  If using ISSUERECLAIM IO model, the the underlying device
   248         *  connected to stream is idled as a result of calling abort and
   249         *  all buffers are ready for reclaim().  The client needs to call
   250         *  {@link #reclaim} to get back the buffers.  However the client will
   251         *  NOT block when calling reclaim() after an abort().
   252         *
   253         *  @b(returns)         IOM_COMPLETED on success, < 0 on failure  
   254         */
   255        Int abort();

   281         *  the mdControl function of the associated mini-driver. The mdControl

   343        /*!
   344         *  ======== issue ========
   345         *  Issue a buffer to the stream.
   346         *
   347         *  This function issues a buffer to the stream for IO. This API is
   348         *  non-blocking.
   349         *
   350         *  Failure of issue() indicates that the stream was not able to accept the
   351         *  buffer being issued or that there was a error from the underlying
   352         *  driver. Note that the error could be driver specific.
   353         *  If issue() fails because of an underlying driver problem
   354         *  {@link #abort} should be called before attempting more I/O through the 
   355         *  stream.
   356         *
   357         *  The interpretation of the logical size of a buffer, is direction
   358         *  dependent.  For a stream opened in OUTPUT mode, the logical size
   359         *  of the buffer indicates the number of minimum addressable units of of 
   360         *  data it contains. 
   361         *
   362         *  For a stream opened in INPUT mode, the logical size 
   363         *  of a buffer indicates the number of minimum addressable units being 
   364         *  requested by the client. In either case, the logical size of the buffer 
   365         *  must be less than or equal to the physical size of the buffer.
   366         *
   367         *  issue() is used in conjunction with {@link #reclaim}. The issue() call 
   368         *  sends a buffer to a stream, and reclaim() retrieves a buffer 
   369         *  from a stream. In normal operation each issue() call is followed by an 
   370         *  reclaim() call.
   371         *
   372         *  Short bursts of multiple issue() calls can be made without an
   373         *  intervening reclaim() call followed by short bursts of reclaim() calls, 
   374         *  but over the life of the stream issue() and reclaim() must be called 
   375         *  the same number of times. The number of issue() calls can exceed the 
   376         *  number of reclaim() calls by {@link #maxIssues}.
   377         *
   378         *  The client argument is not interpreted by IO or the underlying 
   379         *  modules, but is offered as a service to the stream client. All compliant
   380         *  device drivers preserve the value of arg and maintain its association 
   381         *  with the data that it was issued with. arg provides a method for a 
   382         *  client to associate additional information with a particular buffer of 
   383         *  data. The arg is returned during reclaim().
   384         *
   385         *  @param(buf)         buffer pointer
   386         *  @param(size)        size of buffer
   387         *  @param(arg)         application arg
   388         *  @b(returns)         IOM_COMPLETED on success, < 0 on failure  
   389         *  
   390         */
   391        Int issue(Ptr buf, SizeT size, UArg arg);
   392    
   393        /*!
   394         *  ======== reclaim ========
   395         *  Reclaim a buffer that was previously issued by calling {@link #issue}.
   396         *
   397         *  reclaim() is used to request a buffer back from a stream. 
   398         *
   399         *  If a stream was created in OUTPUT mode, then reclaim() returns a
   400         *  processed buffer, and size is zero.  If a stream was opened in
   401         *  INPUT mode, reclaim() returns a full buffer, and size is the number
   402         *  of minimum addressable units of data in the buffer.
   403         *
   404         *  reclaim() calls Sync_wait() with the timeout specified when the 
   405         *  channel was created.  For the default SyncSem, reclaim() blocks
   406         *  until a buffer can be returned to the caller, or until a timeout occurs.
   407         *
   408         *  Failure of reclaim() indicates that no buffer was returned to 
   409         *  the  client. Therefore, if reclaim() fails, the client should 
   410         *  not attempt to de-reference pBuf, since it is not guaranteed to contain
   411         *  a valid buffer pointer.
   412         *
   413         *  reclaim() is used in conjunction with {@link #issue} to operate 
   414         *  a stream. The issue() call sends a buffer to a stream, and 
   415         *  reclaim() retrieves a  buffer from a stream. In normal operation
   416         *  each issue call is followed by an reclaim call.
   417         *
   418         *  Short bursts of multiple issue() calls can be made without an
   419         *  intervening reclaim() call followed by short bursts of reclaim() calls, 
   420         *  but over the life of the stream issue() and reclaim() must be called 
   421         *  the same number of times. The number of issue() calls can exceed the 
   422         *  number of reclaim() calls by {@link #maxIssues}.
   423         *
   424         *  A reclaim() call should not be made without at least one 
   425         *  outstanding issue() call. Calling reclaim() with no 
   426         *  outstanding issue() calls results in an error {@link #E_noBuffersIssued}
   427         *
   428         *  reclaim() only returns buffers that were passed in using issue(). It 
   429         *  also returns the buffers in the same order that they were issued.
   430         *
   431         *  reclaim() returns the size transferred in case of success.
   432         *  It returns zero when an error is caught. In case of timeout, the error 
   433         *  is {@link #E_timeout}.
   434         *
   435         *  @param(pBuf)        returned buffer pointer
   436         *  @param(pSize)       pointer to size of buffer (OUTPUT parameter)
   437         *  @param(pArg)        pointer to client arg. Can be null.
   438         *  @b(returns)         IOM_COMPLETED on success, < 0 on failure  
   439         *  
   440         */
   441        Int reclaim(Ptr *pBuf, SizeT *pSize, UArg *pArg);

   443        /*!
   444         *  ======== read ========
   445         *  Synchronous read command
   446         *
   447         *  An application calls GIO_read to read a specified number of MADUs
   448         *  (minimum addressable data units) from the communication channel.  
   449         *
   450         *  The buf parameter points to a device-defined data structure for
   451         *  passing buffer data between the device and the application. This
   452         *  structure may be generic across a domain or specific to a single
   453         *  mini-driver. In some cases, this parameter may point directly to a
   454         *  buffer that holds the read data. In other cases, this parameter may
   455         *  point to a structure that packages buffer information, size, offset
   456         *  to be read from, and other device-dependent data. For example, for
   457         *  video capture devices this structure may contain pointers to RGB
   458         *  buffers, their sizes, video format, and a host of data required for
   459         *  reading a frame from a video capture device.  Upon a successful read,
   460         *  this argument points to the returned data. 
   461         *
   462         *  The pSize parameter points to the size of the buffer or data
   463         *  structure pointed to by the buf parameter. When the function
   464         *  returns, this parameter points to the number of MADUs read from the
   465         *  device. This parameter is relevant only if the buf parameter points
   466         *  to a raw data buffer. In cases where it points to a device-defined
   467         *  structure it is redundant -- the size of the structure is known to
   468         *  the mini-driver and the application. At most, it can be used for
   469         *  error checking.
   470         *
   471         *  GIO_read returns IOM_COMPLETED upon successfully reading the
   472         *  requested number of MADUs from the device. If an error occurs,
   473         *  the device returns a negative value. For a list of error values,
   474         *  see ti/sybios/io/iom.h. 
   475         *
   476         *  A call to GIO_read results in a call to the mdSubmit function of
   477         *  the associated mini-driver. The IOM_READ command is passed to the
   478         *  mdSubmit function. The mdSubmit call is typically a blocking call,
   479         *  so calling GIO_read can result in the thread blocking.
   480         *
   481         *  @param(buf)         buffer pointer
   482         *  @param(pSize)       pointer to size of buffer (INPUT/OUTPUT parameter)
   483         *  @b(returns)         IOM_COMPLETED on success, < 0 on failure  
   484         */
   485        Int read(Ptr buf, SizeT *pSize);

   530        /*!
   531         *  ======== submit ========
   532         *  Submit an IO job to the mini-driver
   533         *
   534         *  GIO_submit is not typically called by applications. Instead, it is
   535         *  used internally and for user-defined extensions to the GIO module.
   536         *
   537         *  The cmd parameter is IOM_READ, IOM_WRITE, IOM_ABORT or IOM_FLUSH
   538         *  when used internally.  A mini driver may add command codes for 
   539         *  additional functionality.
   540         *
   541         *  The bufp parameter points to a device-defined data structure for
   542         *  passing buffer data between the device and the application. This
   543         *  structure may be generic across a domain or specific to a single
   544         *  mini-driver. In some cases, this parameter may point directly to
   545         *  a buffer that holds the data. In other cases, this parameter may
   546         *  point to a structure that packages buffer information, size, offset
   547         *  to be read from, and other device-dependent data.
   548         *
   549         *  The pSize parameter points to the size of the buffer or data structure
   550         *  pointed to by the bufp parameter. When the function returns, this
   551         *  parameter points to the number of MADUs transferred to or from the
   552         *  device. This parameter is relevant only if the bufp parameter points
   553         *  to a raw data buffer. In cases where it points to a device-defined
   554         *  structure it is redundant -- the size of the structure is known to
   555         *  the mini-driver and the application. At most, it can be used for
   556         *  error checking.
   557         *
   558         *  The appCallback parameter points to either a callback structure that
   559         *  contains the callback function to be called when the request completes.
   560         *  queued request is completed, the callback routine (if specified) is
   561         *  invoked (i.e. blocking).  If the appCallback parameter is NULL, then
   562         *  the call to GIO_submit() will be synchronous and will not return
   563         *  until IO is complete (or an error occurs).  
   564         *
   565         *  GIO_submit returns IOM_COMPLETED upon successfully carrying out
   566         *  the requested functionality. If the request is queued, then a status of
   567         *  IOM_PENDING is returned. If an error occurs, the device returns a
   568         *  negative value. For a list of error values, see ti/sysbios/io/iom.h.
   569         * 
   570         *  A call to GIO_submit results in a call to the mdSubmit function of
   571         *  the associated mini-driver. The specified command is passed to the
   572         *  mdSubmit function.
   573         *
   574         *  @param(cmd)         driver specific packet command
   575         *  @param(buf)         buffer pointer
   576         *  @param(pSize)       pointer to size of buffer (INPUT/OUTPUT parameter)
   577         *  @param(appCallBack) pointer to application callback structure
   578         *  @b(returns)         IOM_COMPLETED on success, < 0 on failure  
   579         */
   580        Int submit(UInt cmd, Ptr buf, SizeT *pSize, AppCallback *appCallback); 
   581        
   582        /*!
   583         *  ======== prime ========
   584         *  Prime an OUTPUT stream instance.
   585         *
   586         *  This API facilitates buffering of an output channel. Consider a
   587         *  task that constantly gets data from input channel and sends to an
   588         *  output channel. To start with it may want to issue buffers to the
   589         *  input channel and output channel to enable double buffering.
   590         *  For an input channel there is no problem. For an output channel however
   591         *  the buffer data is sent out through the peripheral and in the case of a
   592         *  heterogenous system, the data will be sent to the other processor.
   593         *
   594         *  In such cases where the driver cannot handle dummy buffers,
   595         *  IO_prime can be used to make buffers available instantly for
   596         *  reclaim without actually sending the buffers to the driver.
   597         *  This API is non-blocking.
   598         *
   599         *  The primary use of prime() is used when applications want to prime
   600         *  an output channel at startup, without sending data to the driver.
   601         *  This allows them to reclaim and issue in their task.
   602         *
   603         *  Failure of prime() indicates that the stream was not able to accept the
   604         *  buffer being issued  due to un-avaibailibity of IO packets.
   605         *
   606         *  The client argument is not interpreted by IO.
   607         *
   608         *  @param(buf)         buffer pointer
   609         *  @param(size)        size of buffer
   610         *  @param(arg)         app arg
   611         *  @b(returns)         IOM_COMPLETED on success, < 0 on failure  
   612         */
   613        Int prime(Ptr buf, SizeT size, UArg arg);
